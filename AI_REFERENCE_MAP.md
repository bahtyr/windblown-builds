## 2. App architecture

The product is one web app with local JSON data files generated by the scraper.

High-level flow:
1. `packages/scrapper` scrapes wiki pages
2. scraper writes JSON into `apps/web/public/*.json`
3. web app fetches those JSON files client-side
4. deck state, likes, and filters are persisted in `localStorage`

There is no backend/database in the current app.

Important implication:
- most app state is browser-local
- if a feature looks like saved user data, it is probably in `localStorage`
- if content looks like game data, it is probably in `public/*.json`

## 4. State ownership

### Deck state

Primary owner:
- [apps/web/components/deck/DeckContext.tsx](C:/Users/bahti/IdeaProjects/windblown-builds/apps/web/components/deck/DeckContext.tsx)

This is the most important file for:
- saved builds
- active editing session
- duplicate/delete/load/save behavior
- shared deck URL parsing
- deck item identity and ordering

Persistent keys:
- `windblown.deck.v3`: current active deck session
- `windblown.deck.saved.v3`: saved build library

Important helpers in this file:
- `deckId(type, name)`: canonical entity id format
- `makeDeckItem(type, entity)`: convert entity -> deck item
- `resolveSharedDeckFromLocation(...)`: parses `/decks?deck=...`
- `saveDeck(...)`: save/update build behavior
- `hydrateDeckItems(...)`: backfills missing deck item images from entity JSON

Notes:
- deck items use id format `${type}:${name}`
- shared decks are URL-based, not server-backed
- the current library page filters out empty saved decks before rendering

## 11. Known implementation facts

- The app uses local JSON data, not live API data.
- Likes are simple id toggles.
- Saved builds are local only.
- Shared builds are URL-param based.
- The browse page is the real cross-entity UI; old type routes redirect to it.
- The deck library page currently says `Your library`, not `My Builds`.
- `GiftMatchDebug.tsx` currently has inline styles rather than shared CSS.
- `ScrapedEntity` already has optional `video`, but most UI does not use it.
- Image localization in scraper downloads files into `apps/web/public/images/<type>/...`

## 12. Pitfalls for agents

- Do not assume a backend exists.
- Do not assume deck data is centralized outside `DeckContext.tsx`.
- Do not change share-link behavior casually; it is intertwined with deck import on `/decks`.
- If changing entity data shape, check both:
- scraper `packages/scrapper/src/pages/types.ts`
- web `apps/web/lib/types.ts`
- If changing browse filters, check both component state and persisted storage parsing.
- If changing deck item rendering, inspect both library rows and editor drawer; they are separate UIs.
- Ignore `.next` and `node_modules` when searching for source of truth.

## 13. Suggested working pattern for future AI agents

When assigned a task:
1. Read the matching section in this document.
2. Open the ownership file first.
3. Read the relevant test file before editing behavior.
4. Check whether the task touches:
- browser persistence
- generated JSON data
- scraper output
- CSS in a separate file
5. Only then edit.

If the task changes behavior across app and scraper, update both sides in the same pass.
